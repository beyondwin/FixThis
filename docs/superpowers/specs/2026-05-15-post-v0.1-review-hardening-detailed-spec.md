# Spec - Post-v0.1 Review Hardening

Status: Draft
Date: 2026-05-15
Baseline: `v0.1.0..HEAD`
Scope: `:fixthis-mcp`, feedback console JS, CI harness, npm tooling
Related implementation plan: [`../plans/2026-05-15-post-v0.1-review-hardening-implementation.md`](../plans/2026-05-15-post-v0.1-review-hardening-implementation.md)

## Summary

`v0.1.0` 이후 201개 커밋이 들어오면서 event log, session replay, draft workspace, preview fingerprint, console harness, CI가 빠르게 확장됐다. 주요 검증은 통과하지만, 리뷰 결과 몇 가지 안전장치가 아직 production 경로에서 완전히 닫히지 않았다.

가장 큰 위험은 다음 네 가지다.

- event log replay가 wall clock 역행 상황에서 정상 event를 버릴 수 있다.
- server-side fingerprint mismatch 보호가 client body의 `frozenFingerprint`를 신뢰한다.
- draft workspace를 비우는 일부 UI 경로가 schema v2 localStorage workspace를 삭제하지 않아 취소한 annotation이 복구될 수 있다.
- console harness와 CI가 실제 실패/skip/test matrix를 신뢰하기 어렵게 보고한다.

이 문서는 위 결함을 사용자 데이터 정합성, agent handoff 신뢰성, CI 회귀 방지 관점에서 닫는 상세 구현 기준이다.

위 명령은 통과했다. 별도 확인 결과 `npm audit --json`은 direct dev dependency `esbuild@0.21.5`에 moderate advisory를 보고했다. `node scripts/console-harness.mjs --matrix network-outage --viewport mobile-390`은 blocked scenario를 warning으로 skip하면서 `[PASS]`와 exit 0을 반환한다.

## Goals

- event log 파일 정렬과 replay가 `sequenceNumber`를 단일 순서 기준으로 사용한다.
- event log compaction이 production store mutation 뒤 자동 또는 주기적으로 실행된다.
- Save to MCP 경로의 frozen fingerprint는 client body가 아니라 서버가 보관한 preview cache record에서 읽는다.
- draft workspace가 discard/cancel/last-delete 경로에서 localStorage schema v2 workspace까지 삭제한다.
- CI console JS lane이 `scripts/console-tests.json`을 single source of truth로 사용한다.
- console harness가 skip과 pass를 분리해 CLI와 JUnit에 표시한다.
- preview cache disk artifact가 무제한으로 쌓이지 않도록 보존 정책을 둔다.
- `esbuild` advisory는 현재 사용 방식의 위험을 평가하고 업그레이드 또는 명시적 accept 기록으로 닫는다.

## Non-Goals

- Release build 지원. FixThis는 계속 debug-only다.
- View/Flutter target 지원. Compose-only 유지.
- `.fixthis/feedback-sessions/**/session.json`의 public field rename.
- persisted MCP JSON field `items`, `screens`, `itemId`, `screenId`, `targetEvidence`, `sourceCandidates` rename.
- Bridge protocol major version 변경.
- 외부 API 호출, cloud sync, multi-user conflict resolution.

## RH-1 - Event Log Ordering

### Current State

`EventLogWriter.append()`는 파일명을 `"%013d-%010d.jsonl".format(event.epochMillis, event.sequenceNumber)`로 만든다. `EventLogReader.readAll()`은 파일명을 기준으로 정렬한다. `SessionReplayEngine.replayEventsFrom()`은 loop 중 `event.sequenceNumber <= maxSeq`인 event를 idempotent guard로 skip한다.

이 조합은 `System.currentTimeMillis()`가 뒤로 이동하면 다음 순서를 만든다.

1. seq 1, epoch 2000 event 파일 생성.
2. host clock이 1000으로 이동.
3. seq 2, epoch 1000 event 파일 생성.
4. filename 정렬은 seq 2를 먼저 읽는다.
5. replay가 `maxSeq = 2`로 갱신한다.
6. seq 1은 `<= maxSeq`라서 skip된다.

### Requirement

- event replay와 compaction 대상 산정은 decoded `SessionEvent.sequenceNumber` 오름차순을 기준으로 한다.
- 파일명은 backward compatibility 때문에 한 번에 바꾸지 않는다. 먼저 reader/compactor에서 decode-sort를 적용한다.
- 새 event 파일명은 후속 compatibility window 뒤에 sequence-first로 바꿀 수 있다. 이번 작업의 필수 변경은 아니다.
- corrupt event file 하나가 전체 replay를 crash시키는 현재 degrade behavior는 유지한다. 단, 정렬을 위해 decode 과정에서 어떤 파일이 실패했는지 알 수 있는 테스트를 둔다.

### Acceptance Criteria

- seq 1의 epoch이 seq 2보다 큰 event log를 replay해도 두 event가 모두 적용된다.
- compactor threshold가 낮을 때 archive 대상은 wall clock이 아니라 `sequenceNumber`가 가장 낮은 event들이다.
- `EventLogReader.maxActiveSequenceNumberOrNull()`도 filename parse 대신 decoded event 기준으로 max sequence를 계산한다.

## RH-2 - Server-Owned Frozen Fingerprint

### Current State

`FeedbackItemRoutes`는 request body의 `frozenFingerprint`를 `PreviewFeedbackFingerprintCheck`에 넣는다. `FeedbackSessionService.savePreviewFeedbackItemsWithLiveFingerprintMetadata()`는 live recapture를 수행하지만, commit에는 `request.fingerprintCheck.frozenFingerprint`를 그대로 전달한다. `FeedbackDraftService`의 `PreviewFeedbackSaveReservation`은 이미 서버가 보관한 `PreviewRecord`를 들고 있으며, 그 안에 freeze 시점의 `preview.snapshot.screen.fingerprint`가 있다.

즉 서버가 이미 가진 frozen source of truth가 있는데도, mismatch 비교에는 client-provided frozen fingerprint를 사용한다.

### Requirement

- `/api/items/batch` request의 `frozenFingerprint`는 compatibility field로만 남긴다.
- production live save path는 frozen fingerprint를 `reservation.preview.snapshot.screen.fingerprint`에서 읽는다.
- client-provided fingerprint가 서버 preview fingerprint와 다르면 save를 막지 말고 telemetry/debug metadata로만 남긴다. 신뢰 기준은 서버 값이다.
- fallbackScreen path는 preview cache miss 호환 경로다. 이때 frozen fingerprint는 fallback screen의 fingerprint로만 사용할 수 있고, event metadata에 `frozenFingerprintSource = "fallbackScreen"`을 남긴다.
- `forceMismatchOverride=true`는 frozen/current mismatch만 우회한다. recapture 실패, preview not found, invalid item validation은 우회하지 않는다.

### Acceptance Criteria

- client가 `frozenFingerprint = null`을 보내도 cached preview fingerprint가 있고 current fingerprint가 다르면 HTTP 409가 난다.
- client가 current와 같은 bogus `frozenFingerprint`를 보내도 cached preview fingerprint가 다르면 HTTP 409가 난다.
- cached preview fingerprint가 null인 legacy payload는 기존처럼 hard fail하지 않고 `X-FixThis-Fingerprint-Unavailable-Reason`과 event metadata에 unavailable reason을 남긴다.
- forced save event payload에는 `forceMismatchOverride = true`와 fingerprint source/unavailable metadata가 남는다.

## RH-3 - Draft Workspace LocalStorage Cleanup

### Current State

`persistCurrentDraftWorkspaceIfNeeded()`는 `workspaceId`가 있고 item이 1개 이상일 때만 `saveWorkspace()`를 호출한다. workspace가 empty가 되는 경우에는 schema v2 key를 삭제하지 않는다. `resetAnnotationComposerState()`는 empty workspace로 상태를 바꾸고 legacy pending mirror만 지운다. `deletePendingFeedbackItem()`에서 마지막 item을 삭제하면 empty workspace가 메모리에만 반영되고 저장소 cleanup은 호출되지 않는다.

`draftUseCases.resolveDraftBoundary(... discard ...)`에는 올바른 `deleteWorkspace(sessionId, workspaceId)` 흐름이 이미 있다. 문제는 presentation glue가 그 경로를 항상 사용하지 않는 것이다.

### Requirement

- workspace를 clear/discard/cancel할 때는 schema v2 workspace key와 index entry를 삭제한다.
- item count가 1에서 0으로 변하는 delete는 저장 대신 delete를 호출한다.
- legacy pending mirror 삭제와 schema v2 workspace 삭제를 분리해, session switch에서 keep/recover가 필요한 경우는 보존할 수 있어야 한다.
- `setDraftWorkspace()` 또는 그 주변 helper에서 empty workspace transition을 감지해 cleanup을 중앙화한다.

### Acceptance Criteria

- draft item 1개 생성 후 삭제하면 `fixthis.workspace.<session>.<workspace>` key와 index entry가 사라진다.
- draft item 1개 생성 후 cancel하면 reload 뒤 recovery banner가 뜨지 않는다.
- session switch에서 "keep"을 선택한 workspace는 계속 복구 가능하다.
- schema v1 `fixthis.pending.<session>` migration 동작은 유지된다.

## RH-4 - Production Event Log Compaction

### Current State

`EventLogCompactor`는 snapshot-before-archive와 checkpoint를 구현하고, tests도 있다. 그러나 production 생성 경로인 `FixThisTools.defaultFeedbackSessionService()`와 `FeedbackSessionStore` mutation path에는 compactor invocation이 없다. 결과적으로 `.fixthis/feedback-sessions/<session>/events/*.jsonl`은 장시간 console 사용 시 계속 증가한다.

### Requirement

- compactor를 production path에 wiring한다.
- compaction은 store lock 안에서 mutation과 같은 session snapshot을 기준으로 실행되거나, mutation 완료 직후 동일 store의 latest snapshot을 읽어 실행해야 한다.
- threshold는 default 1000 active events로 시작한다.
- compaction 실패는 session mutation 성공을 되돌리지 않는다. 대신 warning/skipped diagnostic으로 노출한다.
- tests에서는 낮은 threshold와 fast durability를 주입할 수 있어야 한다.

### Acceptance Criteria

- default `FixThisTools` store로 한 session에 threshold+N event를 만들면 active `.jsonl` 수가 threshold 이하로 줄고 `events/archive/`와 `events/checkpoint.json`이 생긴다.
- compaction 뒤 fresh store boot replay가 compaction 전 session과 같은 `screens`, `items`, `handoffBatches`, `status`를 만든다.
- compaction 중 archive move 실패가 발생해도 방금 수행한 feedback mutation은 저장되어 있고 다음 mutation이 계속 가능하다.

## RH-5 - CI Uses Console Test Manifest

### Current State

`package.json`은 `console:test:all`을 `scripts/run-console-tests.mjs availability pending beforeunload undo activity preview draft session`으로 정의한다. `scripts/console-tests.json`에는 `draft`와 `session` group이 들어 있다. 그러나 `.github/workflows/ci.yml`은 `node --test` 파일 목록을 직접 나열한다. 이 목록은 현재 draft/session tests를 포함하지 않는다.

### Requirement

- CI는 hard-coded individual JS test list 대신 `npm run console:test:all`을 실행한다.
- fixture/harness unit tests는 `console-tests.json`에 새 group으로 넣거나 별도 `npm run console:harness:test` script로 명명한다.
- `CONTRIBUTING.md`의 required local checks와 CI가 같은 single source를 참조해야 한다.

### Acceptance Criteria

- 새 console test file을 `scripts/console-tests.json`의 group에 추가하면 CI도 자동으로 실행한다.
- `npm run console:test:all`이 local과 CI에서 동일하게 실행된다.
- CI YAML에는 console unit test 파일명이 반복 나열되지 않는다.

## RH-6 - Harness Skip Reporting

### Current State

`scripts/console-harness.mjs`는 `@blocked-pending-impl` scenario를 `{ ok: true, skipped: true }`로 반환한다. CLI summary는 `ok` 기준으로 `[PASS]`를 출력하고, JUnit도 `<testcase />`로 success를 기록한다.

### Requirement

- harness result에는 `status: "passed" | "failed" | "skipped"` 또는 동등한 명시 상태를 둔다.
- CLI는 skipped를 `[SKIP]`으로 출력한다.
- JUnit은 skipped cell을 `<skipped message="..."/>`로 기록하고 `skipped` count를 올린다.
- 명시적으로 `--matrix network-outage`처럼 blocked scenario만 요청한 경우 기본 exit code는 2로 한다. `--allow-skips`가 있으면 exit 0을 허용한다.
- `all` matrix에서 blocked scenario가 포함된 경우는 skipped count를 보고하되 exit code 정책은 CI 의도에 맞게 명시한다. 기본은 skip 허용, nightly required matrix는 `--fail-on-skips`를 사용할 수 있게 한다.

### Acceptance Criteria

- `node scripts/console-harness.mjs --matrix network-outage --viewport mobile-390`은 `[SKIP]`을 출력하고 exit 2를 반환한다.
- 같은 명령에 `--allow-skips`를 붙이면 exit 0이다.
- generated JUnit XML에는 `<testsuite skipped="1">`와 `<skipped>` element가 있다.
- `scripts/console-harness.test.mjs`가 skip XML과 exit policy를 검증한다.

## RH-7 - Preview Cache Disk Retention

### Current State

`PreviewSnapshotCache.put()`은 evicted record list를 반환하지만 `PreviewCaptureService.capturePreview()`는 반환값을 무시한다. saved preview는 `FeedbackDraftService.commitPreviewFeedbackSaveWithMetadata()`에서 `deletePreviewCacheDirectory()`로 삭제된다. 반면 사용자가 저장하지 않고 live preview를 오래 보면 `.fixthis/preview-cache/<session>/<previewId>`가 계속 남는다.

기존 test `capturePreviewRetainsEvictedPreviewCacheDirectoriesForLateScreenshotRequests`는 evicted preview screenshot fallback을 보장한다. 따라서 단순히 memory eviction과 동시에 disk directory를 지우면 현재 UX가 깨진다.

### Requirement

- preview disk retention policy를 별도 class로 둔다.
- memory eviction 직후 즉시 삭제하지 않고, TTL과 per-session max directories를 함께 적용한다.
- default policy:
  - saved preview directory는 기존처럼 save 성공 뒤 즉시 삭제한다.
  - unsaved preview directory는 최소 10분 보존한다.
  - 한 session당 preview directories는 기본 30개를 넘으면 oldest unsaved directory부터 삭제한다.
- `fingerprint-*` probe directory는 현재처럼 recapture finally에서 즉시 삭제한다.
- cleanup 실패는 capture/save 실패로 전파하지 않고 diagnostic log 또는 test-visible callback으로만 남긴다.

### Acceptance Criteria

- 31개 unsaved preview를 만들고 clock을 TTL 이후로 이동하면 oldest directory가 삭제된다.
- TTL 안의 evicted preview screenshot fallback은 계속 동작한다.
- saved preview directory는 save 성공 뒤 즉시 삭제된다.
- `fingerprint-*` directory는 live fingerprint recapture 뒤 남지 않는다.

## RH-8 - esbuild Advisory

### Current State

`package.json`은 direct dev dependency `esbuild: "0.21.5"`를 사용한다. `npm audit`은 `<=0.24.2` 범위에 moderate advisory를 보고하며, fix available은 semver-major `0.28.0`이다.

FixThis는 esbuild build API로 static console bundle을 생성한다. esbuild dev server를 열어 외부 웹사이트가 local dev server 응답을 읽는 공격 경로는 현재 일반 개발 흐름에는 없다. 그래도 direct dependency advisory는 release readiness에서 계속 잡힌다.

### Requirement

- `esbuild`를 현재 ecosystem에서 호환 가능한 최신 버전으로 올리고 bundle output/test를 검증한다.
- major upgrade가 bundle output 또는 Node engine과 충돌하면 `docs/contributing/release-readiness.md`에 explicit temporary accept와 이유를 기록한다.
- advisory check command를 release checklist에 넣을지 결정한다.

### Acceptance Criteria

- 선호 경로: `npm audit --omit=prod` 또는 `npm audit --json`이 `esbuild` advisory를 더 이상 보고하지 않는다.
- fallback 경로: release readiness 문서에 advisory id, affected range, FixThis 사용 방식, revisit date가 기록된다.
- console bundle size budgets와 `node --check`가 계속 통과한다.

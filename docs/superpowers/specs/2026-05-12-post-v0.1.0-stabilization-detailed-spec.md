# Spec — Post-v0.1.0 Stabilization

Status: Draft
Date: 2026-05-12
Baseline: `v0.1.0..HEAD`
Scope: `:fixthis-mcp`, `:fixthis-compose-sidekick`, `:fixthis-compose-core`, console JS, CI
Related implementation plan: [`../plans/2026-05-12-post-v0.1.0-stabilization-implementation.md`](../plans/2026-05-12-post-v0.1.0-stabilization-implementation.md)

## 배경

`v0.1.0` 이후 어노테이션 라이프사이클 하드닝(ALH), 화면 정합성 fingerprinting(SIF), 콘솔 복구 UX, CI 보강이 많이 들어왔다. 현재 전체 Gradle/Node 검증은 통과하지만, 리뷰 결과 일부 핵심 기능이 테스트 전용으로만 연결되어 있거나 런타임 경로에서 의도한 안전장치가 작동하지 않는 결함이 남아 있다.

확인된 현재 상태:

- 전체 Gradle baseline: `spotlessCheck`, `detekt`, 주요 module test, sample assemble, CLI/MCP installDist 통과.
- JS syntax/unit: bundled `app.js` syntax check와 pure Node test 54개 통과.
- `node scripts/build-console-assets.mjs --check`는 동적 build header 때문에 false failure.
- `./gradlew help --warning-mode all --no-daemon`에서 Gradle 10 전 제거 예정 deprecation 경고가 남아 있음.

## 문제 정의

사용자 관점에서 가장 위험한 문제는 "저장했다" 또는 "보호된다"고 기대한 동작이 실제 production console/MCP 경로에서는 보호되지 않는 것이다.

1. Event log는 구현됐지만 `FixThisTools` 기본 store에 writer/reader가 연결되지 않아 실제 MCP/console 런타임에서 write-ahead log, fsync 복구, replay가 작동하지 않는다.
2. Event log compaction은 production에 연결되면 archive 이후 replay가 상태를 잃을 수 있다.
3. Fingerprint mismatch는 frozen 화면과 save 시점의 live 화면을 비교하지 않고, 요청 body 안의 같은 frozen snapshot을 다시 비교하는 구조다.
4. Preview stale 판정은 `frozenAtEpochMillis`가 실제 state에 채워지지 않아 시간/단절 stale이 빈번하게 무효화된다.
5. localStorage pending restore는 pending item 배열만 복구하고 frozen preview context를 복구하지 않아, 복구된 핀이 저장 불가능하거나 새 화면에 잘못 붙을 수 있다.
6. CI가 새로 추가된 JS pure tests와 deterministic console bundle check를 보호하지 않는다.

## 목표

- 실제 MCP/console production 경로에서 event log write-ahead, boot replay, crash recovery가 동작한다.
- Compaction 이후에도 active event replay와 persisted snapshot이 같은 session state를 재구성한다.
- Save to MCP 직전 서버가 live 화면을 재캡처하고 frozen fingerprint와 비교한다.
- Snapshot DTO의 SIF 필드가 sidekick → MCP → domain mapper → persistence 전 구간에서 보존된다.
- Frozen preview에는 freeze 시각과 stale 상태가 항상 기록된다.
- Pending annotation 복구는 item만이 아니라 preview context까지 복구하거나, 복구 불가능하면 명시적으로 폐기/재캡처 UX로 분기한다.
- CI가 console bundle, JS pure tests, Gradle deprecation drift를 잡는다.

## 비목표

- Release build 지원. FixThis는 debug-only 유지.
- View/Flutter target 지원. Compose-only 유지.
- 외부 동기화, cloud persistence, multi-user conflict resolution.
- Bridge protocol major version 변경. Nullable field 추가와 server-side fallback으로 1.x 호환 유지.
- `.fixthis/` session payload의 public JSON field rename.

## STAB-1 — Production Event Log Wiring

### 현재 결함

`FeedbackSessionStore`는 `eventLogWriterProvider`와 `eventLogReaderProvider`를 받을 수 있지만, 기본 production 생성 경로인 `FixThisTools`는 `FeedbackSessionPersistence(FeedbackSessionPaths(projectRoot))`만 연결한다. 따라서 event log 관련 테스트가 green이어도 실제 `fixthis_open_feedback_console`, `fixthis_capture_screen`, `Save to MCP` 경로는 event log 없이 동작한다.

### 요구사항

- `FeedbackSessionPaths`는 session별 event log directory를 canonical path helper로 노출해야 한다.
- `FixThisTools` 기본 store는 다음 세 가지를 같은 `FeedbackSessionPaths` 인스턴스에서 생성해야 한다:
  - `FeedbackSessionPersistence(paths)`
  - `EventLogWriter(paths.eventLogDirectory(sessionId))`
  - `EventLogReader(paths.eventLogDirectory(sessionId))`
- 테스트에서 직접 store를 주입하는 경로는 기존처럼 writer/reader null을 허용한다.
- 실제 HTTP console과 MCP tool call은 동일한 store 인스턴스를 공유해야 한다.

### 수용 기준

- `FixThisTools` 기본 생성자로 feedback session을 열고 item을 저장하면 `.fixthis/feedback-sessions/<sessionId>/events/*.jsonl` 파일이 생긴다.
- 강제 종료 시뮬레이션: event append 성공 후 `session.json` 저장 전 crash를 흉내내도 다음 boot에서 event replay로 item이 복구된다.
- 기존 session에 event directory가 없어도 load/list가 실패하지 않는다.

## STAB-2 — Replay/Compaction Checkpoint Safety

### 현재 결함

현재 replay는 event log가 있으면 `screens`, `items`, `handoffBatches`를 비우고 active `events/*.jsonl`만 적용한다. 반면 compactor는 오래된 event를 `events/archive/`로 옮긴 뒤 `state.json`을 쓴다. Reader는 archive를 읽지 않고, persistence의 canonical file은 `session.json`이다. 이 조합은 compaction이 production에 연결되는 순간 archived events에만 있던 상태를 replay에서 잃을 수 있다.

### 요구사항

- Compaction은 canonical persisted snapshot인 `session.json`을 `FeedbackSessionPersistence.save(session)`로 갱신해야 한다.
- Event log directory에는 checkpoint metadata가 있어야 한다.
- Checkpoint가 없는 session:
  - legacy/full-log 모드로 간주한다.
  - replay는 기존처럼 shell state의 mutable lists를 비우고 모든 active event를 적용한다.
- Checkpoint가 있는 session:
  - `session.json`을 compacted-through snapshot으로 간주한다.
  - replay는 shell state에서 시작하고 `sequenceNumber > compactedThroughSequenceNumber`인 active event만 적용한다.
- Compaction 순서는 durable snapshot/checkpoint 작성 후 archive 이동이어야 한다. Archive 이동 후 snapshot 실패는 금지한다.

### Checkpoint contract

`events/checkpoint.json`:

```json
{
  "schemaVersion": 1,
  "sessionId": "session-id",
  "compactedThroughSequenceNumber": 123,
  "snapshotUpdatedAtEpochMillis": 1778572260000,
  "createdAtEpochMillis": 1778572260000
}
```

### 수용 기준

- 1,200개 event 생성 후 threshold 1,000으로 compaction하면 200개 active event만 남고, boot replay 결과가 compaction 전 session과 byte-equivalent다.
- `events/archive/`만 읽어야 복구되는 상태가 있으면 테스트가 실패해야 한다.
- `checkpoint.json`이 corrupt하면 fail-open으로 잘못된 상태를 만들지 말고 해당 session을 skipped/error로 보고한다.

## STAB-3 — Live Current-Screen Fingerprint Enforcement

### 현재 결함

콘솔은 `addItemsFlow.screen`의 frozen fingerprint를 보낸다. HTTP route는 request body의 `screen?.fingerprint`를 `currentFingerprint`로 넘긴다. 결국 frozen과 current가 같은 request snapshot에서 나오므로 live 화면 변경을 비교하지 못한다. 또한 sidekick은 fingerprint를 null로 두고, MCP `toCapturedScreen()`은 orientation/window/systemUi/fingerprint 필드를 보존하지 않아 check가 자주 short-circuit된다.

### 요구사항

- Save to MCP 직전 서버가 현재 화면을 1회 live recapture해야 한다.
- 비교 대상:
  - frozen: preview freeze 시점의 `SnapshotDto.fingerprint`
  - current: save 시점 live recapture의 `SnapshotDto.fingerprint`
- 둘 중 하나라도 null이면 mismatch를 강제로 통과시키지 말고, `fingerprintUnavailableReason`을 응답/로그에 남긴다. 단, 기존 호환성을 위해 nullable legacy payload는 hard fail하지 않는다.
- `forceMismatchOverride=true`는 mismatch만 우회한다. fingerprint 계산/recapture 예외를 숨기지 않는다.
- `SnapshotDto` ↔ domain mapper는 SIF 필드를 모두 round-trip해야 한다.

### 수용 기준

- freeze 후 app orientation/activity/window mode를 바꾸고 저장하면 HTTP 409 `screen_fingerprint_mismatch`가 반환된다.
- 같은 화면에서 저장하면 200으로 저장된다.
- `forceMismatchOverride=true`로 저장한 item/session에는 override metadata가 event log payload에 남는다.
- `PreviewCaptureService.toCapturedScreen()`이 bridge payload의 orientation, width/height, density, windowMode, systemUiKind, fingerprint를 보존한다.

## STAB-4 — Frozen Preview Timestamp and Staleness

### 현재 결함

`evaluateStale()`은 `state.preview.frozenAtEpochMillis`를 요구하지만 `refreshPreview()`와 `startAddItemsFlow()`가 이 값을 채우지 않는다. `PreviewCaptureService.toCapturedScreen()`도 `capturedAtEpochMillis = 0L`로 저장한다.

### 요구사항

- live preview response를 `state.preview`에 넣을 때 `frozenAtEpochMillis = Date.now()`를 기록한다.
- `addItemsFlow`에도 freeze 시각을 복사한다.
- `SnapshotDto.capturedAtEpochMillis`는 bridge payload가 제공한 capture time이 있으면 보존하고, 없으면 server clock으로 채운다.
- stale 조건:
  - `now - frozenAtEpochMillis > 30_000`
  - bridge connection이 `connected`가 아님
  - orientation/activity drift 감지
- stale 상태에서는 저장 버튼이 사용자에게 recapture/force 선택을 요구해야 한다.

### 수용 기준

- freeze 후 31초 경과 시 stale 배지가 표시된다.
- bridge disconnect 시 다음 status tick에서 stale이 표시된다.
- recapture 후 stale이 해제된다.

## STAB-5 — Pending Recovery Context

### 현재 결함

localStorage restore는 `pendingFeedbackItems` 배열만 복구한다. 그러나 저장 경로는 `addItemsFlow.previewId`, `addItemsFlow.screen`, `addItemsFlow.screenshotUrl`이 필요하다. 이 context가 없으면 `persistPendingFeedbackItems()`가 early return하거나, 사용자가 새 freeze를 시작했을 때 예전 pending pin이 새 화면에 붙을 수 있다.

### 요구사항

- localStorage payload는 item 배열이 아니라 recovery envelope이어야 한다.
- envelope에는 최소한 `sessionId`, `previewId`, `screen`, `screenshotUrl`, `frozenAtEpochMillis`, `items`가 들어간다.
- restore는 자동으로 pending list에 노출하지 않는다. recovery banner에서 사용자가 선택한다:
  - Recover: 같은 session/screen/preview context로 `addItemsFlow` 복원
  - Recapture: items는 보존하되 새 preview에 remap하기 전 사용자 확인
  - Discard: mirror 삭제
- legacy array payload는 "items only"로 읽되 direct save는 막고 recapture/discard만 허용한다.

### 수용 기준

- pending pin 3개 생성 후 reload하면 recovery banner가 뜨고, Recover 선택 시 같은 screenshot과 pins가 복구된다.
- legacy array payload restore 시 Save to MCP가 직접 활성화되지 않는다.
- 새 freeze 시작 시 이전 session/screen의 pending items가 자동으로 섞이지 않는다.

## STAB-6 — Console Asset Determinism

### 현재 결함

`scripts/build-console-assets.mjs --check`는 매 실행마다 현재 epoch/git sha로 build header를 다시 만든 뒤 committed `app.js`와 exact compare한다. 따라서 source module이 동일해도 check가 false-fail할 수 있다.

### 요구사항

- `--check`는 source module body의 stale 여부를 검사해야 한다.
- build header의 epoch/sha는 check에서 정규화하거나, committed header 값을 재사용해 비교해야 한다.
- write mode는 기존처럼 새 epoch/sha를 기록한다.
- CI에 `node scripts/build-console-assets.mjs --check`를 추가한다.

### 수용 기준

- source module 변경 없음: `node scripts/build-console-assets.mjs --check`가 exit 0.
- source module 변경 후 bundle 미갱신: 같은 명령이 exit 1.
- write mode 실행 후 check가 exit 0.

## Rollout 순서

1. CI false failure를 먼저 제거한다(STAB-6). 이후 모든 JS 작업의 guard가 안정화된다.
2. Production event log wiring을 연결한다(STAB-1).
3. Wiring 전에 compaction safety를 고치거나, compactor 호출을 비활성 상태로 유지한 채 STAB-2를 완료한다.
4. Snapshot field preservation을 먼저 고친 뒤 live fingerprint enforcement를 연결한다(STAB-3, STAB-4).
5. Pending recovery envelope UX를 적용한다(STAB-5).
6. CI/Gradle maintenance를 마지막으로 닫는다(STAB-7).

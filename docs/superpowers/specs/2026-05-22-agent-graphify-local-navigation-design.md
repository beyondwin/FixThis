# Agent Graphify Local Navigation Design

Date: 2026-05-22
Status: Ready for user review
Scope: agent-only Graphify navigation guidance, local graph output handling,
and FixThis-specific extraction guardrails

## Summary

FixThis should adopt Graphify as an agent navigation aid, not as product
runtime functionality.

The goal is to help Codex, Claude Code, and similar coding agents orient
themselves in this multi-module Android project before changing code. Graphify
can provide a local code knowledge graph, focused queries, node explanations,
and relationship paths. Those outputs are useful as a map, but they are not a
source of truth for FixThis contracts.

This design keeps Graphify outside the Android runtime, MCP protocol, feedback
handoff schema, source matching algorithm, and release artifacts. The
repository should commit only small operating rules and ignore patterns.
Generated Graphify outputs stay local and are regenerated by each agent when
needed.

## Decisions

1. Graphify is introduced for agent exploration only.
2. `graphify-out/` is local output and must not be committed.
3. The Codex guidance should use Graphify's standard `codex install` section
   as a base, then add a FixThis-specific note.
4. Graph updates are manual by default. Git hook automation is optional and
   should be documented as opt-in.

## Goals

- Reduce the time agents spend locating important modules, call paths, and
  architecture boundaries.
- Give agents a preferred first step for broad codebase questions when a local
  graph already exists.
- Keep local graph generation compatible with FixThis's local-first stance.
- Prevent generated graph data, caches, Android build outputs, and local
  feedback artifacts from feeding back into the graph or into commits.

## Non-Goals

- Do not integrate Graphify into FixThis MCP tools.
- Do not expose Graphify data through `fixthis_read_feedback` or any persisted
  MCP session JSON.
- Do not rename or reinterpret compatibility fields such as `items`,
  `screens`, `itemId`, `screenId`, `targetEvidence`, `targetReliability`, or
  `sourceCandidates`.
- Do not change the source matcher or source-index schema.
- Do not add Graphify to Android app builds, Gradle plugin outputs, release
  packages, or CI gates in this iteration.
- Do not require every contributor or agent to install Git hooks.
- Do not commit generated `graph.json`, `GRAPH_REPORT.md`, `graph.html`, cache,
  memory, wiki, or visualization output.

## Repository Files

The implementation should keep the committed surface small:

```text
.gitignore
.graphifyignore
AGENTS.md
docs/guides/agent-graphify.md
docs/index.md
```

Expected responsibilities:

- `.gitignore` explicitly ignores `graphify-out/`.
- `.graphifyignore` defines FixThis-specific extraction exclusions.
- `AGENTS.md` contains Graphify's standard Codex guidance plus a short
  FixThis addendum.
- `docs/guides/agent-graphify.md` explains local usage and operational
  expectations.
- `docs/index.md` links to the guide if the docs index has an appropriate
  guide section.

No generated Graphify output should be staged.

## Extraction Guardrails

Graphify should not scan local state, generated build output, or its own
outputs. The `.graphifyignore` should cover at least:

```text
.git/
.gradle/
.fixthis/
graphify-out/
build/
**/build/
node_modules/
.DS_Store
*.log
```

It should also exclude local evaluation or fixture paths that can contain
external repositories or generated source matching reports:

```text
.fixthis/eval-fixtures/
build/reports/fixthis-source-matching/
```

The exact ignore file can include additional local IDE or cache directories if
they are already present in this repository's ignore conventions.

## Agent Workflow

For codebase, architecture, or dependency questions:

1. If `graphify-out/graph.json` exists, the agent should start with a focused
   query such as `graphify query "<question>"`.
2. For relationship questions, use `graphify path "<A>" "<B>"`.
3. For a specific symbol, module, or concept, use `graphify explain "<name>"`.
4. If no graph exists and the task benefits from broad orientation, run
   `graphify update .` to create local output.
5. Use `rg`, source reads, docs, and tests to verify any conclusion that will
   affect implementation or product behavior.

`graphify-out/GRAPH_REPORT.md` should be treated as broad context. Agents
should prefer focused `query`, `path`, or `explain` output when possible.

## FixThis Addendum For Agents

The `AGENTS.md` addendum should make these constraints explicit:

- Graphify output is advisory. It can guide exploration, but the authoritative
  sources are repo docs, Kotlin/JS source, Gradle files, and tests.
- For MCP output compatibility, bridge protocol changes, source matching
  confidence, or persisted JSON fields, agents must verify against
  `docs/reference/*` and the implementation before editing.
- Graphify must not be used to justify committing `.fixthis/`,
  `graphify-out/`, Android build output, or local fixture workspaces.
- Graphify failures do not block normal FixThis development. Fall back to
  `rg` and direct source inspection.

## Hook Policy

Manual updates are the default:

```bash
graphify update .
graphify query "where is source matching implemented?"
```

Git hooks are opt-in:

```bash
graphify hook install
graphify hook status
graphify hook uninstall
```

Hooks should not be part of the required project setup because this Android
repository can produce frequent generated-file churn, and Graphify output is a
local aid rather than a project artifact.

## Error Handling

- If `graphify` is missing, agents continue with `rg`, docs, and source reads.
- If `graphify-out/graph.json` is missing, agents may generate it locally but
  should not treat it as required for every task.
- If graph generation is slow or noisy, the first fix should be improving
  `.graphifyignore`, not committing output.
- If Graphify output conflicts with source or docs, the source and docs win.
- If a hook leaves dirty `graphify-out/` files, the files remain local and
  ignored; they are not a reason to change product code.

## Verification

The implementation should verify:

- `git diff --check` passes.
- `graphify --version` runs in the local environment used to write the docs.
- `git status --short` does not show `graphify-out/` or generated Graphify
  artifacts as staged or unstaged tracked changes.
- If practical, `graphify update . --no-cluster` can generate a local graph
  without staging output.
- The docs do not imply Graphify is required for the public CLI, MCP server,
  Android runtime, or release process.

## Open Follow-Up

After this design is approved, the implementation plan should stay docs-first
and narrow: add ignores, install or reproduce the Codex guidance, add the
FixThis addendum, write the guide, and verify that generated output remains
untracked.

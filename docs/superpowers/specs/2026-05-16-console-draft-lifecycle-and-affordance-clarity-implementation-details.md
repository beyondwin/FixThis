tells the implementer *what to type next*; this doc tells them *what to do
when their typing produces a surprise*.

## 2. Policy A enforcement — exact wiring

### 2.1 New state ladder above `DraftLifecycle`

`DraftLifecycle` already exists (`draftWorkspace.js:5-13`). Policy A's
None/Pending/Draft/Saved layer is **derived**, not stored as a new field:

```js
function deriveEditorState(workspace, selection, savedItem) {
  if (savedItem && !workspace) return 'saved';
  if (!selection && !workspace) return 'none';
  if (selection && (!workspace || workspace.lifecycle === 'empty')) return 'pendingTarget';
  if (workspace && workspace.items.length === 0 && !selection) return 'none';
  if (workspace && workspace.lifecycle !== 'empty') return 'draft';
  return 'none';
}
```

Place this in a new file `console/editorState.js` (no DOM, no fetch — pure
derivation) so tests can exercise it without DOM scaffolding. All UI code
(footer renderer, back-nav handler, toast dispatcher) reads from this single
derivation.

### 2.2 Workspace allocation gating

Today the workspace is created when annotate mode is entered. Move
allocation to the first comment-character write.

Touched call sites (from `annotations.js`):

- `enterAnnotateMode` (current behaviour: opens panel, allocates workspace)
  → must allocate only `selection` + open editor, **not** workspace.
- The first non-empty `commentInput.value` write event (`oninput` on
  `#comment`) calls `draftUseCases.ensureWorkspaceForSelection(selection)`.

New use case in `draftUseCases.js`:

```js
function ensureWorkspaceForSelection(selection) {
  if (state.draftWorkspace?.workspaceId) return state.draftWorkspace;
  if (!selection) throw new Error('selection required to allocate workspace');
  const workspace = createFrozenDraftWorkspace({
    workspaceId: nextWorkspaceId(),
    context: selection.context,
    screen: selection.screen,
    screenshotUrl: selection.screenshotUrl,
  });
  dispatch({ type: 'workspace/created', workspace });
  return workspace;
}
```

If the comment is cleared back to `''` (length === 0) and there is only one
draft item:

```js
function maybeFreeWorkspaceOnEmpty(itemId) {
  const items = state.draftWorkspace?.items ?? [];
  if (items.length !== 1) return;
  if (items[0].draftItemId !== itemId) return;
  if (String(items[0].comment || '').length !== 0) return;
  dispatch({ type: 'workspace/freed', reason: 'last-item-empty' });
}
```

This restores the selection but discards the workspace + revision history.
The next non-empty character allocates a fresh workspace.

### 2.3 Storage load-time filter

`draftStorageAdapter.js` — new helper:

```js
function dropEmptyEntries(envelope) {
  let dropped = 0;
  const items = (envelope.items || []).filter(item => {
    const comment = String(item.comment || '');
    if (comment.length === 0) { dropped += 1; return false; }
    return true;
  });
  if (dropped > 0) {
    console.info(`[draft-recovery] skipped ${dropped} empty-comment entries`);
  }
  return { ...envelope, items };
}
```

Apply at every load entry point:
- `loadWorkspacesForSession(sessionId)`
- `loadPendingState(sessionId)` (in `pendingPersistence.js`)
- `restorePendingState(sessionId)` (in `historyPendingDedupe.js` recovery
  path)

If the envelope becomes empty after filtering, the load function returns
`null`/`[]` exactly as it would for "no recovery." Callers do not need new
branches.

### 2.4 Server validation

`FeedbackItemRoutes.kt` save handlers — add validation before delegating to
`AnnotationWorkflow`:

```kotlin
private fun SaveSnapshotRequest.validateComments() {
    val empties = items.count { it.comment.isBlank() }
    if (empties > 0) {
        throw PreviewFeedbackRequestValidationException(
            "Cannot save annotation with empty comment (${empties} item(s)).",
            code = "empty-comment",
        )
    }
}
```

Wire into both `decodeSavePreviewFeedbackItemsBody` consumers
(`FeedbackItemRoutes.kt:39, 117`). The mapping to
`FeedbackConsoleHttpException` already exists at line 176; add a new branch:

```kotlin
text.startsWith("Cannot save annotation with empty comment") ->
    422 to "empty-comment"
```

Why server-side and client-side: client gates *creation*, server rejects
*persistence*. Two doors; one bug class.

## 3. Boundary protocol — trigger routing

The lifecycle hardening spec (`2026-05-15-console-draft-session-lifecycle-hardening-detailed-spec.md`)
established a single boundary protocol fire site. This spec extends it by
routing every trigger from design §2 through it, including the
previously-uncovered Pending-Target row.

### 3.1 Trigger registry

New file `console/boundaryTriggers.js`:

```js
const Trigger = Object.freeze({
  SESSION_SWITCH: 'sessionSwitch',
  SESSION_CREATE: 'sessionCreate',
  SESSION_DELETE: 'sessionDelete',
  SCREEN_SWITCH: 'screenSwitch',
  NEW_CAPTURE: 'newCapture',
  ELEMENT_CLICK: 'elementClick',
  ESCAPE_KEY: 'escapeKey',
  BROWSER_REFRESH: 'browserRefresh',       // beforeunload
  TAB_CLOSE: 'tabClose',                   // beforeunload
  ROUTE_CHANGE: 'routeChange',
  SERVER_DISCONNECT: 'serverDisconnect',
  RECONNECT: 'reconnect',
  BRIDGE_MISMATCH: 'bridgeMismatch',
  ACTIVITY_DRIFT: 'activityDrift',
  INACTIVITY: 'inactivity',
  EDITOR_BACK: 'editorBack',
});
```

Every call site that currently triggers a session-changing action imports
this registry and calls a single resolver:

```js
async function resolveTrigger(trigger, payload) {
  const state = deriveEditorState(currentWorkspace(), currentSelection(), currentSavedItem());
  const verdict = boundaryPolicy(trigger, state, payload);
  // verdict ∈ {'pass', 'silentDiscard', 'discardWithToast', 'boundaryDialog',
  //            'beforeunloadGuard', 'staleRevalidate'}
  return applyVerdict(verdict, trigger, payload);
}
```

`boundaryPolicy(trigger, state, payload)` is the table in design §2 turned
into a switch. Implemented as a pure function in `console/boundaryPolicy.js`
with one unit test per cell of the matrix (15 × 3 = 45 cases).

### 3.2 Call-site migration

| Trigger | Today's call site | Migration |
|---|---|---|
| `SESSION_SWITCH` | `history.js` session row click handler | Wrap click in `await resolveTrigger(Trigger.SESSION_SWITCH, {targetSessionId})`. |
| `SESSION_CREATE` | `history.js:296` (`+` icon) | Same. |
| `SESSION_DELETE` | New trash-icon handler (Section 5.1) | Calls resolver before deletion. |
| `ESCAPE_KEY` | `shortcuts.js:11-19` | Already routes through `cancelAddItemsFlow` / `clearSelection`. Replace with `resolveTrigger(Trigger.ESCAPE_KEY, {})`. |
| `BROWSER_REFRESH` / `TAB_CLOSE` | New `beforeunload` listener in `main.js` | Verdict: `beforeunloadGuard` returns truthy when state === draft (browser shows native warning); silent for Pending. |
| `ELEMENT_CLICK` | `annotations.js` preview click handler | Silent target swap for Pending, target-change confirm for Draft. |
| `SERVER_DISCONNECT` / `RECONNECT` | `connection.js` FSM transitions | Resolver triggers status surface registry rather than direct UI toggle. |
| `EDITOR_BACK` | New back-button handler (Section 7) | Resolver decides discard vs dialog vs immediate-back. |
| `SCREEN_SWITCH` / `NEW_CAPTURE` | `preview.js` snapshot update handler | Pending → discard + toast; Draft → preserved (anchored). |

### 3.3 Toasts

`resolveTrigger` is also the place that fires transparency toasts (design
§4). Toast text is interpolated from the trigger:

function renderInspectorFooter(state, ctx) {
  const root = document.getElementById('inspectorFooter');
  root.dataset.editorState = state;
  switch (state) {
    case 'none':
      root.hidden = true;
      root.replaceChildren();

## 5. `StatusSurfaceRegistry` — module shape

### 5.1 Module skeleton

New file `console/statusSurfaceRegistry.js`:

```js
const SurfaceClass = Object.freeze({
  MODAL: 'modal',
  MODAL_CANVAS: 'modalCanvas',
  BANNER: 'banner',
  INLINE: 'inline',
  BADGE: 'badge',
  TOAST: 'toast',
  PILL: 'pill',
});

const Priority = Object.freeze({
  CRITICAL: 0,
  ERROR: 1,
  WARNING: 2,
  INFO: 3,
});

function createStatusSurfaceRegistry() {
  const state = {
    surfaces: new Map(),    // surfaceId → { class, priority, element, content, timer }
    visible: new Map(),     // class → Set<surfaceId>
  };

  function show(surfaceId, { surfaceClass, priority, content, autoDismissMs }) { /* ... */ }
  function hide(surfaceId) { /* ... */ }
  function suspend(surfaceClass) { /* hides all of class without removing */ }
  function resume(surfaceClass) { /* restores suspended */ }
  function size(surfaceClass) { return state.visible.get(surfaceClass)?.size ?? 0; }

  return { show, hide, suspend, resume, size };
}
```

Singleton instance created in `consoleApp.js`; passed to surface modules via
existing port-injection pattern.

### 5.2 Coordination rules (enforced inside `show`)

```js
function show(surfaceId, opts) {
  // Rule 1: modal exclusion
  if (opts.surfaceClass === SurfaceClass.MODAL ||
      opts.surfaceClass === SurfaceClass.MODAL_CANVAS) {
    suspend(SurfaceClass.INLINE);
    suspend(SurfaceClass.BADGE);
  }

  // Rule 2: stack limits
  const limit = STACK_LIMIT[opts.surfaceClass];   // toast=3, banner=1, modal=1
  while (size(opts.surfaceClass) >= limit) {
    const oldest = oldestOfClass(opts.surfaceClass);
    hide(oldest);
  }

  // Rule 3: priority preemption — a CRITICAL banner replaces a WARNING banner
  // ...

  // Render + register
}
```

Auto-dismiss is implemented via `setTimeout`, cleared on `hide`.

### 5.5 Disconnect / reconnect choreography

Old disconnect path (current): `connection.js` directly toggles overlay +
banner + badge + lock bar.

New path:

```js
function onDisconnect(reason) {
  registry.show('canvasBlockedOverlay', {
    surfaceClass: SurfaceClass.MODAL_CANVAS,
    priority: Priority.ERROR,
    content: { headline: 'Device disconnected', detail: 'Reconnecting…', retry: true },
  });
  if (hasDirtyDraft()) {
    registry.show('stalenessBanner', {
      surfaceClass: SurfaceClass.BANNER,
      priority: Priority.WARNING,
      content: '⚠ 1 unsaved draft preserved locally',
    });
  }
  // No direct .hidden writes on inline surfaces — registry suspends them.

function onReconnect() {
  registry.hide('canvasBlockedOverlay');
  registry.show('previewStaleBadge', {
    surfaceClass: SurfaceClass.BADGE,
    priority: Priority.INFO,
    content: 'Connection restored · refreshing preview',
    autoDismissMs: 2000,
  });
  registry.resume(SurfaceClass.INLINE);
  registry.resume(SurfaceClass.BADGE);
}
```

      btn.hidden = true;
    }

## 8. Migration & backward compatibility

### 8.1 LocalStorage compatibility

- Schema is unchanged. Existing entries with empty comments are silently
  ignored at load time (Section 2.3) and will fall out of localStorage via
  the existing eviction in the next save cycle.
- If a user has a draft with empty comment that they care about (edge: they
  selected an element, typed nothing, and reloaded), it is lost. Acceptable
  per design §1.

### 8.2 Server compatibility

- The new validation returns a typed `422` with `empty-comment` action. The
  console maps it through `FeedbackConsoleHttpException` already.
- External agents calling the save route directly will see this new error
  only if they previously sent empty comments. No agent in the current
  surface does — verified via `grep -rn 'comment.*=' fixthis-cli/` and
  `fixthis-mcp/src/main/kotlin/.../tools/` (no producer sends empty).

### 8.3 Test fixtures

Console regression tests under
`fixthis-mcp/src/main/console/__tests__/` may have fixtures that exercise
empty-comment paths. Audit:

```bash
grep -rln 'comment.*=.*""\|"comment":\s*""' \
  fixthis-mcp/src/main/console/__tests__/
```

Each such fixture either:
- Migrates to a non-empty comment (if the test wasn't really about
  emptiness).
- Becomes a Policy A negative test (`expect(...).toThrow('empty-comment')`).

### 8.4 No bridge/MCP changes

The Android sidekick, bridge protocol, and MCP tool signatures are
untouched. `BridgeProtocol.VERSION` does not bump.

## 11. Open implementation questions

These are technical decisions to make during implementation. They do not
affect the design's semantic invariants.

1. **`commentInput` event throttling.** `oninput` fires per keystroke;
   workspace allocation should happen on the first non-empty transition,
   not every keystroke. Trivial guard inside `ensureWorkspaceForSelection`
   (early-return if workspace exists).
2. **Animation timing.** Use existing `--transition-fast` / `--transition-slow`
   custom properties if defined in `styles.css`; otherwise pick 160ms / 300ms
   defaults and call them out in a follow-up.
3. **Toast container element.** Add once near `#error` in `index.html`:
   `<div id="toastContainer" class="toast-container" aria-live="polite"
   aria-atomic="false"></div>`. Multiple toasts append; oldest removed by
   registry on overflow.
4. **`localStorage` quota awareness.** `dropEmptyEntries` reduces footprint;
   no new quota concerns expected.
5. **Reduced-motion respect.** Toasts and overlay fades read
   `prefers-reduced-motion: reduce` and disable transitions. Pre-existing
   convention in this codebase if applicable; otherwise add.

## 12. Rejected alternatives

- **Server-only empty-comment validation (without client gating).** Rejected
  — the user-visible bug (Pending Target lifecycle confusion) is client-
  side. Server validation alone leaves the local recovery-dedup race.
- **Reform `DraftLifecycle` to include `pendingTarget`.** Rejected —
  `DraftLifecycle` is a workspace-internal enum; Pending Target is the
  absence of workspace. Mixing them would require ad-hoc null-handling at
  every consumer. Deriving above is simpler.
- **Persist Pending Target for refresh recovery.** Rejected per design §2
  trigger 8. Reintroduces the dedup edge case that Policy A eliminates.
- **Replace the four-button footer in place (keep IDs, change behaviour).**
  Rejected — preserves the misleading IDs and forces conditional code in
  every test. A clean state-dependent slot is less risky.
- **Use a third-party toast library.** Rejected — adds bundle size, breaks
  the local-first / zero-runtime-dep posture. Registry is < 200 LOC.

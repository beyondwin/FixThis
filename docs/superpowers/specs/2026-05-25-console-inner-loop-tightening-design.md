# Console Inner Loop Tightening Design

Date: 2026-05-25
Status: Ready for user review
Scope: Reduce daily maintainer friction in the console inner loop by
automating the JS rebundle→reload cycle and verifying that Kotlin server
restart preserves in-progress browser state. Out of scope: Gradle build time
itself, JVM hot-swap, new dev transports, external-user onboarding.

## Summary

The console inner loop has two daily-frequency pain surfaces today:

1. **JS edits** require a manual `node scripts/build-console-assets.mjs` run,
   a manual browser reload, and a separate `--check` invocation before PR. The
   `console-build-meta.json` sidecar must stay in lockstep with the bundle or
   `--check` fails in CI.
2. **Kotlin edits** require `bash scripts/restart-console.sh`, which is fine
   on its own, but the browser then loses any in-progress draft / annotation /
   tool-mode state because the page is forced to reload via a cache-bust query
   string.

This spec tightens the JS cycle by adding an esbuild watch pipeline that
keeps the bundle, source map, and build-meta atomically in sync and pushes a
reload signal over the existing SSE channel when the maintainer is running
with `--console-assets-dir`. It also verifies that the existing SSE/polling
reconnect path preserves in-progress browser state across a Kotlin restart,
and adds a single status chip so the maintainer can see at a glance which
server build the console is currently connected to.

Gradle build time itself is left alone. JVM hot-swap is left alone. No new
dev transport is introduced — the existing `/api/events` SSE channel carries
the reload signal.

## Goals

- A maintainer editing `fixthis-mcp/src/main/console/*.js` should not have to
  manually rebundle, manually refresh the browser, or remember to run
  `--check` before pushing.
- `console-build-meta.json` must remain consistent with the bundle byte-for-byte
  after every watch-mode rebuild, so the existing `--check` gate passes
  without an extra step.
- After a `bash scripts/restart-console.sh`, the browser should reconnect
  automatically and any in-progress draft / annotation / tool-mode state that
  was already persisted to `.fixthis/` or `localStorage` should be present
  after reconnect.
- The console should display which server build it is currently connected to,
  so the maintainer can confirm a restart actually delivered new server code.

## Non-Goals

- Reducing Gradle `installDist` time itself. Gradle's daemon, parallel, and
  caching flags are already enabled in `gradle.properties`; further reduction
  is a separate concern.
- JVM hot-swap, JRebel-like tooling, Gradle `--continuous` daemons, or a
  long-running dev daemon that gracefully restarts the server process. These
  are cost-asymmetric for the actual maintainer pain they remove.
- A unified "dev command" wrapper script that multiplexes JS watch, Gradle
  watch, and process management. Not requested, and risks the same
  theoretical-slot-filling that was rejected during brainstorming.
- A new dev transport such as WebSocket. The existing SSE channel
  (`/api/events`) is sufficient for the reload signal.
- Any changes that affect packaged-JAR mode. Watch-mode reload and the
  related signaling apply only when the server is run with
  `--console-assets-dir`. The packaged JAR continues to serve the inlined
  assets exactly as today.
- External-user DX (golden path smoke, `fixthis init`, `fixthis doctor`,
  `fixthis install-agent`). This spec is about the maintainer inner loop.
- New MCP tools, persisted JSON schema changes, or compact-handoff format
  changes.

## Design Principles

1. Reuse the existing SSE channel and `--console-assets-dir` mode. Do not
   introduce a new transport or a new operating mode.
2. The watcher's only job is to keep build artifacts atomically in sync. The
   reload signal is the Kotlin server's responsibility once it observes a
   bundle change.
3. The packaged-JAR serving path is the production contract. Watch-mode
   behavior must be additive and gated on `--console-assets-dir`.
4. Verify before extending. Track 2 starts by confirming the existing
   reconnect+hydration path holds across a Kotlin restart and only adds code
   where a gap is found.
5. The maintainer must be able to tell, without reading logs, which server
   build the browser is currently talking to.

## Track 1: JS Watch Pipeline

### Current Problem

`scripts/build-console-assets.mjs` is invoked manually after each JS edit.
The maintainer must then refresh the browser manually. `console-build-meta.json`
is regenerated by the same script; if the maintainer rebundles without
regenerating the meta sidecar (e.g. by running esbuild ad hoc), the
`--check` mode at `scripts/build-console-assets.mjs --check` fails the next
time it is invoked, typically only in CI. Three manual steps per JS edit, one
silent foot-gun.

### Components

#### 1.1 `build-console-assets.mjs --watch`

Add a `--watch` flag to `scripts/build-console-assets.mjs`. When set:

- Use esbuild's native `context().watch()` API on the existing build config so
  the bundler stays resident and rebuilds incrementally on file changes under
  `fixthis-mcp/src/main/console/`.
- On every successful rebuild, write `app.js`, `app.js.map`, and
  `console-build-meta.json` atomically (write-then-rename for each file) into
  `fixthis-mcp/src/main/resources/console/`. The same atomic write strategy
  used by the existing one-shot build must be reused; do not introduce a new
  write path.
- The watch-mode artifacts must be byte-identical to the one-shot
  `FIXTHIS_BUNDLE_REPRODUCIBLE=1` artifacts for the same source tree, so the
  existing `--check` gate passes immediately after stopping the watcher.
- On rebuild failure (syntax error, missing `// @requires`, contract symbol
  missing), print the error and keep watching. Do not write partial artifacts.
  Do not exit.
- On `SIGINT`, dispose the esbuild context and exit cleanly.

#### 1.2 Server-side bundle change detection

`FeedbackConsoleServer.kt` (when started with `--console-assets-dir`) polls
`console-build-meta.json` mtime in the asset directory at a fixed interval
(default 500 ms). When the mtime advances:

- Read the new `console-build-meta.json` and extract the bundle hash.
- Emit a new SSE event on `/api/events`:
  `event: console-assets-changed`, `data: {"buildHash":"<hash>","at":"<iso-time>"}`.
- The poll loop runs only when `consoleAssetsDir != null`. Packaged-JAR mode
  is unaffected.

The mtime polling approach is chosen over a Node→Kotlin HTTP signal so that
the watcher process and the server process remain decoupled and either can be
restarted independently. The asset directory is already read by the server on
every request in dir-mode, so disk-watch cost is negligible.

#### 1.3 Browser-side reload handling

`fixthis-mcp/src/main/console/sse.js` (or the appropriate SSE consumer)
subscribes to the new `console-assets-changed` event. When received:

- Compare the incoming `buildHash` to `window.FixThisConsoleConfig.buildMeta.buildHash`.
  If equal, ignore (defensive — avoid reload loops if the server replays).
- If different, call `location.reload()`. Browser state persistence is
  handled by the existing `localStorage` / `.fixthis/` contracts; Track 2
  verifies those paths.
- Reload is gated on a `window.FixThisConsoleConfig.devReloadEnabled` flag
  populated only when the server is in `--console-assets-dir` mode, so the
  packaged-JAR build cannot trigger reloads.

### Data Flow

```
JS file edit
  └─> esbuild watch rebuilds → atomic write of app.js + app.js.map + console-build-meta.json
        └─> Kotlin server mtime poll observes new console-build-meta.json
              └─> SSE emits console-assets-changed { buildHash }
                    └─> Browser SSE consumer compares hash, calls location.reload()
                          └─> Reloaded page re-hydrates from localStorage + GET /api/sessions
```

### Error Handling

- Bundler error during watch: print to watcher stdout, keep watching, do not
  write partial artifacts. Server has nothing to observe → no reload.
- `console-build-meta.json` mtime advances but the file is partial or
  malformed: server logs at WARN, skips the SSE emit, retries on the next
  poll. Atomic write in 1.1 should prevent this path, but the server must
  tolerate transient inconsistency without crashing the SSE loop.
- Browser receives `console-assets-changed` but reload fails (rare): the
  event is fire-and-forget. The maintainer can manually refresh; no state is
  lost beyond what a normal reload loses.

### Acceptance Criteria

- Editing any file under `fixthis-mcp/src/main/console/` while
  `node scripts/build-console-assets.mjs --watch` is running causes the
  browser tab (opened against a `--console-assets-dir` server) to reload
  within 2 seconds of save, without any manual action.
- Stopping the watcher and running
  `node scripts/build-console-assets.mjs --check` exits 0 — the watch-mode
  artifacts pass the reproducibility check.
- Restarting only the Kotlin server (without restarting the watcher) does not
  cause spurious reloads on connect: the initial SSE event must not be a
  `console-assets-changed` event for the already-current bundle hash.
- The packaged JAR (no `--console-assets-dir`) emits no
  `console-assets-changed` events and the browser never reloads itself.

### Tests

- `scripts/build-console-assets.test.mjs` (new): runs `--watch` in a temp
  dir, edits a fixture JS file, asserts that artifacts are atomically
  rewritten and that `--check` against the resulting tree exits 0.
- `FeedbackConsoleServerTest.kt`: with a temp `consoleAssetsDir`, advance
  `console-build-meta.json` mtime and assert a `console-assets-changed` SSE
  event is emitted with the new bundle hash. Without `consoleAssetsDir`,
  assert no such event fires.
- Browser-side: a `consoleApp.test.js` (or the appropriate existing harness)
  case that simulates the SSE event and asserts `location.reload()` is
  invoked only when the incoming hash differs from the current one.

## Track 2: Restart Reconnect Verification

### Current Problem

`bash scripts/restart-console.sh` kills the server, rebuilds, and restarts
on the same port. The browser's open SSE connection drops; existing
client-side reconnect should reattach. Drafts and annotations are persisted
to `.fixthis/` and partially to `localStorage`, so in principle reconnect
should hydrate them — but this has never been explicitly verified across a
restart, and the maintainer has no visual confirmation that the new server
build is the one currently serving them.

### Components

#### 2.1 Restart-reconnect integration test

A new integration test in `:fixthis-mcp:test` (or a `scripts/` harness if a
browser is required) exercises this sequence:

1. Start a console server pointing at a temp `.fixthis/` workspace.
2. Open an SSE subscription and a session-polling client.
3. POST a draft annotation so it persists to `.fixthis/`.
4. Stop the server process.
5. Start a new server process on the same port pointing at the same
   workspace.
6. Assert: the SSE/polling client reconnects within N seconds without
   manual intervention, and the previously persisted draft is present in
   the next `GET /api/sessions` response.

If the test reveals a gap (e.g. some draft state lives only in browser
memory and is lost on reload), the gap is fixed by persisting that state to
`localStorage` or to `.fixthis/`. No new persistence mechanism is invented
— use whichever existing mechanism already covers similar state.

#### 2.2 Server-build status chip

The console already inlines `console-build-meta.json` at serve time as
`window.FixThisConsoleConfig.buildMeta`. Extend it to include a server-side
build identifier (Kotlin module build hash, already available via the
existing server-version surface).

Add a status chip to the console top bar that displays one of:

- `connected · build sha=<short-hash>` — normal steady state
- `reconnecting…` — SSE drop detected, retry in flight
- `connected · build sha=<new-hash>` — same display, but with a brief
  visual flash when the hash transitions, so the maintainer can confirm
  the restart delivered a different server build

The chip is read-only; clicking it does nothing. The intent is signal, not
control.

### Data Flow

```
Maintainer runs restart-console.sh
  └─> Kotlin server process dies → SSE clients see drop
        └─> Browser reconnect loop reattaches when server is back
              └─> Status chip transitions: connected → reconnecting → connected (new sha)
                    └─> Existing hydration logic refills sessions / drafts from .fixthis/
```

### Error Handling

- Reconnect never succeeds (port still held, server failed to start):
  status chip stays in `reconnecting…` indefinitely. This matches today's
  behavior; this spec does not add a recovery UI for this case.
- Server-side build hash unavailable: the chip falls back to
  `connected` without an sha suffix. Do not block render on the chip.

### Acceptance Criteria

- The restart-reconnect integration test passes deterministically: drafts
  persisted before restart are visible after restart without manual reload.
- After `bash scripts/restart-console.sh` against an open browser tab, the
  status chip transitions through `reconnecting…` and lands on a new
  `build sha=…` value matching the freshly-built server.
- No new persistence file or schema field is introduced; any hydration gap
  found by the test is closed by reusing existing `.fixthis/` or
  `localStorage` mechanisms.

### Tests

- `RestartReconnectIntegrationTest.kt` (new) — covers the sequence in 2.1.
- `consoleApp.test.js` case (or equivalent) — given an SSE drop event,
  asserts the status chip transitions to `reconnecting…` and back to
  `connected · build sha=<new>` once a `server-version` event is received.

## Sequencing

Track 1 then Track 2.

Track 1 must land first because the status chip in 2.2 depends on the
`window.FixThisConsoleConfig.buildMeta` inlining path that Track 1 hardens
(atomic write of `console-build-meta.json` is what makes the chip value
trustworthy under fast iteration).

Within Track 1, the implementation order is 1.1 → 1.2 → 1.3. The watcher is
self-contained and testable without the server change. The server's mtime
poll and SSE emit can be developed against a manually-touched
`console-build-meta.json`. Browser handling lands last so it can be tested
end-to-end against the real watch loop.

Within Track 2, 2.1 lands before 2.2 because the integration test will
surface any draft-hydration gaps that the chip alone would mask.

## Verification Strategy

- Each track's acceptance criteria are guarded by automated tests as listed.
- Manual smoke (documented in `CONTRIBUTING.md` updates below): start the
  watcher and a `--console-assets-dir` server, edit a JS file, confirm the
  browser reloads within 2 seconds. Then run `restart-console.sh` and
  confirm the status chip transitions through the expected states with no
  lost draft.
- `node scripts/build-console-assets.mjs --check` must pass against the
  watch-mode artifacts (regression against the existing CI gate).
- `BridgeProtocolVersionSyncTest` and the existing console asset tests
  (`ConsoleAssetRoutesTest`, `ConsoleFeedbackItemRoutesTest`) must continue
  to pass unchanged.

## Risks And Mitigations

- **Reload loops**: a bug that re-emits `console-assets-changed` for the
  same hash would loop the browser. Mitigation: hash equality check in 1.3,
  plus a server-side guard that does not emit if the last-emitted hash
  equals the newly read one.
- **Watcher dies silently**: if the esbuild watcher crashes, the maintainer
  may not notice until edits stop reloading. Mitigation: watcher prints a
  heartbeat line on each successful rebuild; absence is the signal. No
  daemon supervision is added.
- **mtime granularity on macOS/Linux**: fast successive saves can produce
  identical mtimes. Mitigation: watcher always rewrites
  `console-build-meta.json` (changing its mtime via rename) on every
  successful build, even if content is unchanged. Server polls on mtime,
  not content hash, so the SSE guard (hash equality) is the dedup point.
- **Restart reconnect race**: if the browser reconnects faster than the
  server finishes initializing, the first request may 5xx. Mitigation:
  existing client retry-with-backoff covers this. The new test must assert
  reconnect succeeds within a generous window (e.g. 10 s), not on the
  first attempt.

## Acceptance For This Spec

- All Track 1 and Track 2 acceptance criteria are met by automated tests
  and pass in CI.
- Documentation in `CONTRIBUTING.md` and
  `docs/reference/feedback-console-contract.md` reflects the new dev loop.
- `node scripts/build-console-assets.mjs --check` continues to pass with no
  argument-shape changes.
- No public output schema or persisted JSON field is renamed or removed.
- The packaged-JAR serving path is observably unchanged: no new SSE event
  types appear for non-dir-mode servers, and the browser never reloads
  itself in production mode.
